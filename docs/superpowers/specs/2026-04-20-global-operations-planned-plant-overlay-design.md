## Problem Statement

The `/global-operations` page needs to ingest newly planned plants from the `plant-created` WebSocket feed and show them on both the globe and 2D map without disturbing existing site behavior.

Each WebSocket-delivered plant should:

- read `plantId`, `longitude`, `latitude`, `plantName`, `country`, `province`, and `city`
- appear as a new green tag point on the map
- persist across page refreshes in the same browser
- use its own lightweight popup template
- keep all existing map points and existing popups unchanged

The new popup must only show:

- plant name
- country / province / city
- fixed status `规划中`
- simulated supporting metrics
- a `继续规划` button

Clicking `继续规划` must navigate to:

`http://10.180.40.166/#/workspace/info/home?workflowId=${plantId}&tab=forward`

## What Makes This Cool

This turns the map from a static global operations mural into a live planning surface.

The nice part is not the dot itself. The nice part is the split behavior:

- existing operating plants keep their rich operations popup
- newly created planning plants get a different visual language, a different popup, and a direct handoff into the planning workflow

That gives the page a real "system in motion" feel without rewriting the current operations experience.

## Constraints

- Existing global site points must remain exactly as they are today.
- Existing global site popup behavior must remain exactly as it is today.
- The new feature is frontend-only for now.
- Browser refresh persistence is required, but cross-browser or cross-user persistence is not.
- The WebSocket payload is assumed to already contain the required fields; the frontend does not synthesize missing plant identity fields.
- The current map stack is `MapLibre GL JS`, with one GeoJSON source for existing sites and click handling already bound to that source.

## Premises

1. WebSocket plants are a second map entity type, not a mutation of the existing `GlobalSitePoint` model.
2. Planned plants should be stored in browser `localStorage`, then rehydrated on page load before or alongside incoming WebSocket events.
3. Planned plants must render on their own source/layer pair so the old source and old click interactions stay stable.
4. Planned plants need a popup template separate from the existing `SitePopupCard`.
5. The `继续规划` button is the only call to action required for planned plants in this iteration.

## Approaches Considered

### Approach A: Fold WebSocket Plants Into Existing GlobalSitePoint

Treat WebSocket plants as if they were normal operational sites. Convert them into `GlobalSitePoint`, merge them into the existing `points` array, and branch inside the existing popup template based on a marker flag.

Pros:

- fewest files
- quickest path to visible map dots
- least new plumbing at the data boundary

Cons:

- mixes planning semantics into operations entities
- makes popup behavior conditional and fragile
- increases the chance of breaking old site clicks and old popup rendering
- gets worse the next time a third map point type is introduced

### Approach B: Recommended, Add A Dedicated Planned Plant Overlay

Create a dedicated planned-plant frontend model, browser persistence helper, WebSocket-to-overlay adapter, and separate MapLibre source/layer set for planned plants. Keep the old source, old popup, and old click handlers intact.

Pros:

- keeps existing site behavior untouched
- gives new points their own styling and popup template
- makes local persistence straightforward
- keeps product boundaries obvious in code

Cons:

- introduces a few small supporting files
- requires the map scene to manage two point systems instead of one

### Approach C: Full Unified Map Entity Protocol

Refactor the global map around a generalized map-entity layer where both existing operations sites and planned plants conform to a shared discriminated union, with a common rendering and popup dispatch pipeline.

Pros:

- cleanest long-term architecture
- easiest to extend with more map entity types later
- best if the page is becoming a general geospatial workspace

Cons:

- too invasive for this feature
- touches more existing code than necessary
- high risk of collateral regressions in current operations behavior

## Recommended Approach

Choose Approach B.

This is the right cut because the product requirement is not "all points are the same now." The requirement is "old points keep acting like operations points, new WebSocket plants act like planning points." That is a separation problem, not a conditional rendering trick.

Approach B gives us the cleanest product boundary and the lowest regression risk. It preserves the current site source, current site popup, and current site click path exactly as they are, while adding a second overlay for planned plants.

## Open Questions

- None that block implementation. The action URL, persistence scope, payload fields, color, and popup content are all specified.

## Success Criteria

The feature is done when all of the following are true:

1. A `plant-created` WebSocket message containing `plantId`, `longitude`, `latitude`, `plantName`, `country`, `province`, and `city` creates a new map point.
2. The new point is green on both globe and 2D modes.
3. Existing points render and behave exactly as before.
4. Clicking an existing point opens the current operations popup, unchanged.
5. Clicking a planned-plant point opens the new simplified popup, not the existing popup.
6. The new popup shows only the planning identity block, fixed `规划中` status, simulated metrics, and the `继续规划` button.
7. Clicking `继续规划` opens:
   `http://10.180.40.166/#/workspace/info/home?workflowId=${plantId}&tab=forward`
8. Refreshing the page restores previously received planned plants from browser storage.
9. Duplicate WebSocket events for the same `plantId` do not create duplicate planned points.

## Distribution Plan

Existing frontend deployment pipeline covers this.

No new distribution surface is needed. This ships as part of the current `frontend` app and follows the same local dev, build, and Kubernetes deployment path already in place.
